《编写可读代码的艺术》笔记

Aug 21, 2022

Ch.1 代码应当易于理解


💡 可读性基本定理
代码的写法应当使别人理解它所需要的时间最小化

  • 并不是单纯越短越好
  • 可读性与其他指标并不冲突
    代码容易理解，可以将其引向好的架构
  • 经常思考代码是否易于他人理解，最难的部分

Part.1 表面层次的改进

Ch.2 名字传递信息

  • 选择专业的词
  • 避免泛泛的词
  • 具体代替抽象
  • 使用前/后缀带上附加信息
  • 决定名字的长度
    • 作用域大，名字更长比较好
  • 格式传递信息

Ch.3 命名

  • 范围选区
    • 上下极限： max/ min
    • 包含范围： first/ last
    • 含首排尾： begin/ end

Ch.4 审美

  • 用方法来整理不规则的东西
    如 React 中的 warning , invariant 方法
  • 把声明按块组织起来，把代码分成「段落」
  • 选择一个有意义的顺序，并且始终保持

Ch.5 该写什么样的注释

  • 不需要
    • 不要为了注释而注释
    • 不要给坏的名字加注释
      好代码>坏代码+好注释
  • 需要
    • 加入「导演评论」
    • 为代码瑕疵写注释
      • TODO 未处理的事情
      • FIXME 已经有问题的代码
      • …
    • 给常量加注释
  • 克服「作者心理阻滞」
    1. 不管心理想什么，先写下来
    2. 读一下这段注释，看看可否改进
    3. 重复 step2

Ch.6 写出言简意赅的注释

  • 尽量避免使用代词
  • 声明代码的高层次意图
  • 嵌入的注释解释参数
    （这点使用默认值，和类型信息可以补充？
  • 用含义丰富的词？

Part.2 简化循环和逻辑

Ch.7 简化控制流


💡 把条件、循环以及其他对控制流的改变做的越「自然」越好。运用一种方式使读者不用停下来重读你的代码

  • 条件参数顺序
  • if/else 语句块顺序
  • do/while 循环未必是好
  • 提前的 return
  • 最小化嵌套
    • 提早返回减少嵌套
    • 减少循环内的嵌套

Ch.8 拆分超长表达式

  • 解释变量
    1. 拆分长的表达式
    2. 通过简单的名字描述子表达式，让代码文档化
    3. 帮助阅读时，理解代码中的主要概念

  • 德摩根定理操作逻辑表达式

    $$
    \mathtt{not}\ (a\ \mathtt{or}\ b\ \mathtt{or}\ c)\Leftrightarrow(\mathtt{not}\ a)\ \mathtt{and}\ (\mathtt{not}\ b)\ \mathtt{and}\ (\mathtt{not}\ c)\
    \mathtt{not} (a\ \mathtt{and}\ b\ \mathtt{and}\ c)\Leftrightarrow(\mathtt{not}\ a)\ \mathtt{or}\ (\mathtt{not}\ b)\ \mathtt{or}\ (\mathtt{not}\ c)
    $$

  • 慎用短路逻辑

Ch.9 变量和可读性

  • 减少变量
    变量越多，越难全部跟踪它们的动向
    • 移除无法价值 temp
    • 减少控制流变量
    • 减少中间结果

  • 缩小作用域

    变量作用于越大，需要跟踪它的动向越久

    • 用局部作用域，降格
      • 静态方法
      • 拆类
    • 定义下移
  • 尽量使用常量
    变量改变越频繁，越难跟踪它的当前值

Part.3 重新组织代码

Ch.10 抽取不相关的子问题


💡 自顶向下或者自底向上编程结合，重要的是移除并解决单独子问题

  • 简单说就是抽象

    所谓工程学就是关于把大问题拆分成小问题再把这些问题的解决方案放回一起

  • 警惕过犹不及，会造成读代码时的不连贯性

Ch.11 专注一件事

强调读写代码时的连贯性

Ch.12 Thinking to Coding

用自然语言描述，然后转化成代码

Ch.13 Less Code


💡 最好读的代码就是没有代码

  • 质疑和产分你的需求

  • 保持小代码库

    一旦代码写好后，程序员往往不情愿删除它，因为它代表很多实际的工作量。删掉它可能以为这承认在上面所花的时间就是浪费。不要这么想！这是一个有创造性的邻域——摄影家、作者和电影制作人也不会保留他们所有的工作。

  • 保持对标准库及其 API 的熟悉

Part.4 精选话题

Ch.14 测试和可读性


💡 测试应当具有可读性，以便其他程序员可以舒服地改变或者增加测试


💡 选择一组最简单的输入，它能完整地使用被测试代码

  • 让测试更可读
    对使用者隐去不重要的细节，以便突出更重要的细节；同时，写测试的代码越少，越会倾向于添加测试
    • 创建最小的测试声明
    • 实现定制的「微语言」
  • 让错误消息更可读
  • 选择好的测试输入
    • 简化输入值
    • 多组小测试>单组大测试
  • 为测试函数命名
    别怕长，反正不常用
  • 别走太远
    • 牺牲真实代码的可读性，仅为了使其能被测试
    • 着迷于 100% 覆盖率
    • 让测试成为产品开发的阻碍
